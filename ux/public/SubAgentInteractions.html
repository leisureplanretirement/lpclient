Sub-Agent Interaction Patterns

GreenAcres · Agent Project · MultiPart Orchestration

Task Definitions

Each task corresponds to a specialised sub-agent class. They share the same ISubAgent interface but receive different system prompts and different views of the shared ModelContext.

Task
Sub-Agent Class
When Used
Purpose
Task Z
QuerySufficiencySubAgent
First query only
Gatekeeper. Determines whether the user has provided enough information (income, expenses, savings) to run a meaningful simulation. Responds with SUFFICIENT or NOT SUFFICIENT plus a description. If not sufficient, it replies directly to the user and clears the model context so the next attempt still counts as first time.
Task A
ToolCallSubAgent
Both paths
Unconditional tool caller. Forces an MCP RunSimulation tool invocation regardless of context. Uses the ToolCallOnly system prompt with old tool responses elided. Does not analyse the result — returns only the raw [FunctionCall, FunctionResult, AssistantText] messages for the caller to append to context.
Task B
AnalyzeToolResponseSubAgent
Subsequent queries only
Speculative analyser (prior result). Runs in parallel with Tasks A and C. Analyses the previous simulation result without waiting for a new tool call. Its output is used only if Task C concludes no new tool call is needed.
Task C
NeedToolResponseSubAgent
Subsequent queries only
Decision gate. Inspects prior context and the new user query to decide whether a fresh simulation is required. Returns one of: call_tool, reuse, answer_without_tool, or ask_clarifying_question. Receives a compacted summary of prior tool responses (not the raw 400 KB payload) to keep its prompt concise.
Task D
AnalyzeToolResponseSubAgent
Both paths
Fresh-result analyser. Launched after Task A completes. Analyses the current simulation result and produces the user-facing textual response. Uses the AnalysisOnly system prompt; tool invocation is disabled. Output is delivered via a deferred logger so only the winning task's text reaches the user.
Task E
ToolCallValidationSubAgent
First query only
Validator. After Task A runs, an independent LLM call checks the generated tool-call JSON for date consistency, schema compliance, and fidelity to the user's stated inputs. Returns TOOL REQUEST VALID or TOOL REQUEST INVALID. Triggers a corrective retry loop (up to 2 attempts) on failure.

Shared Infrastructure

🗂
ModelContext

The single shared conversation history. All sub-agents read a read-only snapshot via FetchCompactedContext(), which can inject a different system prompt and elide large tool payloads. Only the orchestrator writes back to ModelContext (after Task A completes). Calling Clear() resets to the initial system prompt, used when Task Z says NOT SUFFICIENT.

📋
DeferredDialogLogger

Tasks B and D each hold their reply in a deferred buffer. Only once the winning task is determined does that buffer flush to the real dialogLogger — ensuring the user sees exactly one response.

📅
DatePreprocessor (step 0)

On the first query, before any task runs, a separate LLM call rewrites the user's message to make all ages and dates explicit (e.g. "retire at 65" becomes "retire at 65 (2041-03-01)"). On subsequent queries the user text is added directly to context.

Pattern 1 — First Query

The first query triggers a validation-heavy flow. The goal is to ensure the user's intent is clear before running the simulation, and to verify the generated tool request is correct after.

Key points — First Query

  • 1
    Task Z and Task A are launched simultaneously. Task Z is awaited first — if the user hasn't provided enough information, the flow stops immediately and Task A's result (when it arrives) is discarded.
  • 2
    Once Task A succeeds, its three messages (FunctionCall, FunctionResult, AssistantText) are written permanently into ModelContext.
  • 3
    Task E and Task D run in parallel. Task E validates the tool request; Task D generates the user-facing analysis.
  • 4
    If Task E returns INVALID, KickOffAED() injects Task E's critique messages into the context for correction guidance, then re-runs Tasks A, E, and D. Up to 2 retry attempts are made.
  • 5
    Task D's deferred response is flushed to the user only after Task E confirms validity (or retries are exhausted).

Pattern 2 — Subsequent Queries

From the second query onward, validation (Tasks Z and E) is skipped. The system bets on being able to reuse the prior simulation result and races three tasks in parallel to minimise latency. 

  User Query
      │
      ▼  (added directly to ModelContext as User message)
      │
      ├──────────────────────────────────────────────────────────────────┐
      │ (parallel launch — all three start immediately)                  │
      ▼                          ▼                          ▼
  ┌──────────────────┐   ┌──────────────────┐   ┌──────────────────────┐
  │  Task A            │   │  Task B            │   │  Task C               │
  │  ToolCallSubAgent│   │  AnalyzeTool     │   │  NeedToolResponse    │

                                 │

  (before C+B decide) ──────────────► Task D wins immediately  ✓

Key points — Subsequent Queries

  • 1
    All three tasks start in parallel. Task A runs the simulation unconditionally — it does not wait to hear Task C's verdict.
  • 2
    As soon as Task A finishes, its results are packaged into a new context snapshot and Task D is launched. Task A's messages are not written into ModelContext yet — only the winning path does that.
  • 3
    If Task C concludes reuse/answer_without_tool and Task B has already finished, Task B wins and Task A/D's work is discarded. The shared ModelContext is not updated with the unwanted tool call.
  • 4
    If Task D finishes before C+B reach a joint conclusion, Task D wins immediately — the race is over.
  • 5
    Task C receives a summarised view of prior tool responses (not the raw 400 KB payload) to keep its decision prompt concise. The summary reads like: "Tool returned simulation results successfully (1 investor, 2 accounts, ~30 years of projections)."
  • 6
    Only the winning task's deferred logger is flushed to the user. Both Task B and Task D write to independent DeferredDialogLogger instances so their outputs don't collide.
Trade-off: Task A always runs on subsequent queries, even when the result is ultimately discarded (Task B wins). This wastes one MCP call but minimises perceived latency: the new simulation result is ready immediately if Task C decides it is needed.

Agent Detail Cards

Task Z
QuerySufficiencySubAgent

System prompt — SystemPrompt-QuerySufficiency.txt

Context view — Only the system message + current user message. Ignores shared ModelContext entirely.

Output — JSON: { "judgement": "SUFFICIENT" | "NOT SUFFICIENT", "description": "…" }

On NOT SUFFICIENT — Logs description to user, calls ModelContext.Clear(), returns true (firstTime stays true next call).

Failure behaviour — Defaults to SUFFICIENT to avoid blocking valid queries.

Task A
ToolCallSubAgent

System prompt — SystemPrompt-ToolCallOnly.txt

Context view — FetchCompactedContext(forceElideAllResponses: true). Old tool responses replaced with "Data removed for brevity".

Output — Three ChatMessage objects: FunctionCall, FunctionResult, AssistantText. Tool request JSON stored in ChatQueryRepository.

Retry — Inner loop of up to 2 attempts for transient provider errors (ArgumentOutOfRangeException).

Task B
AnalyzeToolResponseSubAgent (prior result)

System prompt — SystemPrompt-AnalysisOnly.txt

Context view — FetchCompactedContext() without Task A's new messages — the prior tool result only.

Tools — None (ChatToolMode.None).

Output — User-facing natural-language response stored in a DeferredDialogLogger. Flushed only if Task B wins the race.

Task C
NeedToolResponseSubAgent

System prompt — Built dynamically by SystemPromptManager.GetToolNeededRawPrompt(). Includes compacted prior context + full MCP schema.

Context view — Single System message (entire prompt is the system message). No conversation history passed.

Output — JSON: { "action": "call_tool" | "reuse" | "answer_without_tool" | "ask_clarifying_question" }

Key optimization — Large tool payloads (400 KB+) are summarised to ~150 chars before being embedded, preventing prompt overload.

Task D
AnalyzeToolResponseSubAgent (fresh result)

System prompt — SystemPrompt-AnalysisOnly.txt

Context view — FetchCompactedContext() after Task A's messages have been appended — includes the new simulation result.

Tools — None (ChatToolMode.None). Forbidden from invoking tools; must base answer on the RunSimulation output.

Output — User-facing natural-language response in a DeferredDialogLogger. Flushed if Task D wins.

Task E
ToolCallValidationSubAgent

System prompt — Built dynamically by SystemPromptManager.GetToolCallValidationRawPrompt(). Includes the ToolCallOnly system prompt (for reference), the user query, the generated JSON, and the MCP schema.

Context view — Single System message. Completely isolated from ModelContext.

Output — Plain text containing TOOL REQUEST VALID or TOOL REQUEST INVALID.

On INVALID — Task E's messages are passed to KickOffAED() as correction guidance for the next attempt.

Failure behaviour — Technical exceptions return IsSuccess = true (cannot invalidate without a valid response).
